## VisĂŁo geral

Apps conversacionais tratam cada linha do usuĂˇrio como uma **nova execuĂ§ĂŁo do flow** com o **mesmo id de sessĂŁo**. A CrewAI oferece helpers para histĂłrico de mensagens, classificaĂ§ĂŁo opcional de intenĂ§ĂŁo, tracing adiado, pontes para UI e um REPL local `flow.chat()` para flows conversacionais.

| Conceito | ImplementaĂ§ĂŁo |
|---------|----------------|
| Id de sessĂŁo | `kickoff(session_id=...)` â†’ `inputs["id"]` â†’ `state.id` |
| Linha do usuĂˇrio | `kickoff(user_message=...)` acrescenta em `state.messages` antes do grafo rodar |
| Fim do turno | `FlowFinished` sĂł para **esta execuĂ§ĂŁo**; o chat segue no prĂłximo `kickoff` |
| Trace da sessĂŁo | `ConversationalConfig(defer_trace_finalization=True)` + `finalize_session_traces()` |

## APIs de turno

Use **`flow.kickoff(user_message=..., session_id=...)`** ou **`flow.handle_turn(...)`** para cada mensagem de usuĂˇrio em REST, WebSocket, testes e UIs customizadas. Use **`flow.chat()`** quando quiser um loop de chat local no terminal para um `Flow` conversacional.

| API | Uso |
|-----|-----|
| `kickoff(user_message=..., session_id=...)` | Cada mensagem do usuĂˇrio |
| `handle_turn(message, session_id=...)` | Wrapper ergonĂ´mico de um turno para `Flow` conversacional |
| `chat()` | REPL local no terminal para `Flow` conversacional |
| `kickoff_async(...)` | Mesmos parĂ˘metros; entrada async nativa |
| `ask()` | Prompt bloqueante **dentro** de um passo (wizard, esclarecimento) |
| `@human_feedback` | Aprovar/rejeitar **saĂ­da de um passo** â€” nĂŁo a prĂłxima linha do chat |
| `ChatSession.handle_turn(...)` | Camada de transporte sobre `kickoff` (SSE / WebSocket) |

## InĂ­cio rĂˇpido

```python
from uuid import uuid4

from crewai.flow import (
    ChatState,
    ConversationalConfig,
    Flow,
    listen,
    or_,
    persist,
    router,
    start,
)
from crewai.flow.persistence import SQLiteFlowPersistence


class SupportFlow(Flow[ChatState]):
    conversational_config = ConversationalConfig(
        default_intents=["order", "help", "goodbye"],
        intent_llm="gpt-4o-mini",
        defer_trace_finalization=True,
    )

    @start()
    def bootstrap(self):
        if not self.state.session_ready:
            self.state.session_ready = True
        return "ready"

    @router(bootstrap)
    def route(self):
        # last_intent definido em prepare_conversational_turn quando default_intents estĂˇ setado
        return self.state.last_intent or "help"

    @listen("order")
    def handle_order(self):
        reply = "Seu pedido estĂˇ a caminho."
        self.append_message("assistant", reply)
        return reply

    @listen("help")
    def handle_help(self):
        reply = "Como posso ajudar?"
        self.append_message("assistant", reply)
        return reply

    @listen("goodbye")
    def handle_goodbye(self):
        reply = "AtĂ© logo!"
        self.append_message("assistant", reply)
        return reply

    @persist(SQLiteFlowPersistence("support.db"))
    @listen(or_(handle_order, handle_help, handle_goodbye))
    def finalize(self):
        return self.state.model_dump()


session_id = str(uuid4())
flow = SupportFlow()

flow.kickoff(user_message="Onde estĂˇ meu pedido?", session_id=session_id)
flow.kickoff(user_message="E as devoluĂ§Ăµes?", session_id=session_id)
flow.finalize_session_traces()  # um link de trace para o chat inteiro
```

## Ciclo de vida do turno

Cada `kickoff` com `user_message` executa este pipeline:

1. **`_configure_conversational_kickoff`** â€” mescla `session_id` / `user_message` em `inputs`, aplica `ConversationalConfig`, habilita tracing adiado quando configurado.
2. **RestauraĂ§ĂŁo de estado** â€” se `inputs["id"]` existe e `@persist` estĂˇ configurado, carrega o snapshot mais recente.
3. **`FlowStarted`** â€” emitido apenas no primeiro turno da sessĂŁo adiada.
4. **`prepare_conversational_turn`** â€” acrescenta a mensagem do usuĂˇrio em `state.messages`, define `last_user_message`, limpa `last_intent`, classifica opcionalmente quando `intents` / `default_intents` + `intent_llm` estĂŁo definidos.
5. **ExecuĂ§ĂŁo do grafo** â€” `@start` â†’ `@router` â†’ handlers `@listen`.
6. **Fim da execuĂ§ĂŁo** â€” `flow_finished` por turno e finalizaĂ§ĂŁo de trace sĂŁo **ignorados** com adiamento; `Agent.kickoff()` / crews aninhados tambĂ©m nĂŁo fecham o batch pai.

Os handlers devem chamar **`append_message("assistant", reply)`** para que o prĂłximo turno inclua a resposta do assistente. A linha do usuĂˇrio jĂˇ Ă© salva no kickoff â€” nĂŁo acrescente de novo nos handlers.

## `ConversationalConfig` (padrĂµes em nĂ­vel de classe)

Defina na subclasse de `Flow` como `conversational_config: ClassVar[ConversationalConfig | None]`.

| Campo | PadrĂŁo | PropĂłsito |
|-------|---------|-----------|
| `default_intents` | `None` | RĂłtulos de outcome para classificaĂ§ĂŁo automĂˇtica antes do kickoff |
| `intent_llm` | `None` | Modelo para classificaĂ§ĂŁo (obrigatĂłrio quando hĂˇ intents) |
| `interactive_prompt` | `"You: "` | Prompt para `kickoff(interactive=True)` |
| `interactive_timeout` | `None` | Timeout por linha no modo interativo |
| `exit_commands` | `exit`, `quit` | Palavras que encerram o modo interativo |
| `defer_trace_finalization` | `True` | Manter um batch de trace aberto entre turnos |

Sobrescreva por kickoff com `intents=` e `intent_llm=`.

## PadrĂµes de roteamento de intenĂ§ĂŁo

### A. PrĂ©-classificar via `ConversationalConfig` (mais simples)

Defina `default_intents` e `intent_llm`. Cada kickoff classifica antes do `@router`; leia `self.state.last_intent` em `route()`.

### B. Classificar dentro do `@router` (prompts mais ricos)

Defina `default_intents=None` para o kickoff sĂł acrescentar a mensagem. Em `route()`, chame `classify_intent` com prompt ou descriĂ§Ăµes customizadas:

```python
@router(bootstrap)
def route(self):
    intent = self.classify_intent(
        self._routing_prompt(self.state.last_user_message),
        ("GREETING", "ORDER", "RESEARCH", "GOODBYE"),
        llm=self.conversational_config.intent_llm or "gpt-4o-mini",
    )
    self.state.last_intent = intent
    return intent
```

Use **`@listen("RESEARCH")`** (ou similar) para passos com `Agent.kickoff()` e ferramentas â€” nĂŁo `LLM.call()` puro â€” quando precisar de pesquisa web ou uso multi-etapa de tools.

## Quando o flow termina mas o usuĂˇrio continua conversando

`FlowFinished` significa que **esta execuĂ§ĂŁo do grafo** terminou. A conversa segue com outro `kickoff` e o mesmo `session_id`. `@persist` restaura `messages`, flags e contexto.

**PadrĂŁo de persistĂŞncia:** prefira `@persist` em um **Ăşnico passo terminal** (por exemplo `finalize`) em vez de na classe `Flow` inteira. Persist em nĂ­vel de classe salva apĂłs cada mĂ©todo; `load_state` usa a linha mais recente, que pode ser snapshot no meio da execuĂ§ĂŁo e perder atualizaĂ§Ăµes dos handlers no mesmo turno.

NĂŁo use `@human_feedback` para linhas de chat de follow-up, a menos que um humano precise aprovar uma saĂ­da especĂ­fica antes de exibi-la.

## `Flow` conversacional (experimental)

  `RouterConfig`, `ConversationState`, o grafo embutido + helpers) vive em
  `crewai.experimental` e pode mudar de formato antes de graduar. Fixe a
  versĂŁo do CrewAI se depende de comportamento especĂ­fico e acompanhe o
  changelog para mudanĂ§as quebradoras. Feedback / issues bem-vindos.

Habilite o grafo conversacional definindo `conversational = True` em uma subclasse de `Flow`. O `Flow` base passa a expor um grafo embutido `@start` / `@router` / `converse_turn` / `end_conversation`, gerencia `state.messages`, dirige o LLM de roteamento e mantĂ©m o batch de trace aberto entre os turnos. VocĂŞ escreve as **rotas customizadas**; o framework cuida do resto.

Use isto quando quiser um chat multi-turno com router LLM e handlers por rota sem cablar o ciclo de vida na mĂŁo. Use `Flow[ChatState]` (o padrĂŁo de mais baixo nĂ­vel acima) quando precisar de controle total.

### Exemplo rĂˇpido

```python

    ConversationConfig,
    ConversationState,
    RouterConfig,
)


ROUTER_LLM = LLM(model="gpt-4o-mini")


@ConversationConfig(

    llm=ROUTER_LLM,

)
class SupportFlow(Flow[ConversationState]):
    conversational = True

    @listen("INTERNET_SEARCH")
    def handle_internet_search(self) -> str:
        """Fresh web research, current news, real-time lookups."""
        ...
        self.append_assistant_message(reply)
        return reply

    @listen("CREWAI_DOCS")
    def handle_crewai_docs(self) -> str:
        """Look up the CrewAI documentation for framework/API questions."""
        ...
        self.append_assistant_message(reply)

flow = SupportFlow()
try:

    flow.handle_turn("Pesquise na web por notĂ­cias de IA.")  # roteia para INTERNET_SEARCH
```

Para um chat local no terminal, use `chat()`:

```python

```

`chat()` envolve `handle_turn()` em um REPL, sai com `exit` / `quit`, ignora linhas em branco por padrĂŁo e chama `finalize_session_traces()` quando a sessĂŁo termina.

### `ConversationConfig`

Decorador de classe que anexa os defaults de chat por classe.

| Campo | PadrĂŁo | PropĂłsito |
|-------|--------|-----------|
| `system_prompt` | `slices.conversational_system_prompt` (i18n) | System message usado pelo `converse_turn` embutido. Passe `""` para desativar totalmente. |
| `llm` | `None` | LLM de conversa (usado pelo `converse_turn` e como fallback do router). |
| `router` | `None` | `RouterConfig` para roteamento por LLM. Sem ele, o flow sempre cai em `converse`. |
| `answer_from_history_prompt` | padrĂŁo do framework | System message para a rota opcional `answer_from_history`. |
| `answer_from_history_llm` | `None` | Habilita o atalho `answer_from_history` quando definido. |
| `intent_llm` | `None` | LLM para o caminho legado `intents=`/`default_intents`. |
| `default_intents` | `None` | Labels de outcome para prĂ©-classificaĂ§ĂŁo legada. |
| `visible_agent_outputs` | `None` | `"all"` ou lista de nomes de agentes cujos `append_agent_result()` devem virar mensagens pĂşblicas. |
| `defer_trace_finalization` | `True` | MantĂ©m um Ăşnico batch de trace aberto entre chamadas de `handle_turn()`. |

### `RouterConfig` e o catĂˇlogo de rotas auto-gerado

```python
RouterConfig(
    prompt="Enquadramento de domĂ­nio opcional (polĂ­tica, voz, persona).",
    response_format=MyRoute,        # opcional; auto-gerado caso contrĂˇrio
    llm=ROUTER_LLM,                  # usa ConversationConfig.llm como fallback
    routes=["INTERNET_SEARCH", "CREWAI_DOCS"],   # opcional; inferido dos listeners
    route_descriptions={
        "INTERNET_SEARCH": "Sobrescreve a docstring sĂł desta rota.",
    },
    default_intent="converse",       # usado quando a chamada ao LLM falha ou nĂŁo hĂˇ LLM
    fallback_intent="converse",      # usado quando o LLM retorna rota invĂˇlida
    intent_field="intent",
)
```

O prompt do router Ă© montado automaticamente. Para cada rota o framework escolhe a descriĂ§ĂŁo nesta precedĂŞncia:

1. `RouterConfig.route_descriptions[label]` â€” override explĂ­cito.
2. `Flow.builtin_route_descriptions[label]` â€” texto canĂ´nico do framework para `converse`, `end`, `answer_from_history` (otimizado para o LLM de routing).
3. Primeira linha nĂŁo vazia da docstring do handler `@listen(label)`.
4. Vazio (a rota aparece no catĂˇlogo sem descriĂ§ĂŁo).

Na prĂˇtica, **adicionar uma rota Ă© `@listen("X")` + uma docstring de uma linha**:

```python
@listen("INTERNET_SEARCH")
def handle_internet_search(self) -> str:
    """Fresh web research, current news, real-time lookups."""
    ...
```

â€¦e o LLM de routing vĂŞ:

```
Routes:
- CREWAI_DOCS: Look up the CrewAI documentation for framework/API questions.
- INTERNET_SEARCH: Fresh web research, current news, real-time lookups.
- converse: Ordinary chat, follow-ups, summaries, clarificationsâ€¦
- end: User signals the conversation is finished (goodbye, exit, done).
```

`RouterConfig.prompt` Ă© para **enquadramento de domĂ­nio** (persona do assistente, regras de negĂłcio, voz). O catĂˇlogo de rotas Ă© auto-gerado â€” nĂŁo liste rotas em `prompt`; elas vĂŁo sair de sincronia assim que vocĂŞ adicionar um handler.

### Rotas embutidas

| Rota | Handler | PropĂłsito |
|------|---------|-----------|
| `converse` | `converse_turn` | Handler de chat padrĂŁo. Chama `ConversationConfig.llm` com o system prompt + histĂłrico canĂ´nico. |
| `end` | `end_conversation` | Define `state.ended = True` e emite uma resposta de encerramento. |
| `answer_from_history` | `answer_from_history_turn` | Opcional. Cai aqui quando `ConversationConfig.answer_from_history_llm` estĂˇ definido e a mensagem pode ser respondida sĂł pelo histĂłrico. |

VocĂŞ pode sobrescrever qualquer uma definindo um handler com o mesmo nome na subclasse.

### SemĂ˘ntica de `handle_turn()`

`flow.handle_turn(message)` roda um turno:

1. Reseta o tracking por execuĂ§ĂŁo (`_completed_methods`, `_method_outputs`) para o grafo re-rodar â€” sem isso, chamadas repetidas de `kickoff` na mesma instĂ˘ncia dariam curto-circuito no turno 2+ porque `Flow.kickoff_async` trata `inputs={"id": ...}` como restauraĂ§ĂŁo de checkpoint.
2. Anexa a mensagem do usuĂˇrio em `state.messages`, define `current_user_message` / `last_user_message`. `last_intent` Ă© **preservado do turno anterior** para que o LLM de routing possa usĂˇ-lo como sinal.
3. Roda `conversation_start` â†’ `route_conversation` â†’ o handler `@listen` escolhido.
4. O router grava sua decisĂŁo em `state.last_intent` (visĂ­vel para o contexto de routing do prĂłximo turno).
5. Se seu handler retornou uma string e ainda nĂŁo chamou `append_assistant_message`, `handle_turn` anexa para vocĂŞ.

VocĂŞ tambĂ©m pode chamar `flow.kickoff(user_message=..., session_id=...)` diretamente â€” a mesma lĂłgica de reset/run Ă© acionada. `handle_turn` Ă© o wrapper ergonĂ´mico.

### `chat()` para REPLs locais

`flow.chat()` Ă© o wrapper de terminal pronto para uso em cima de `handle_turn()`:

```python
flow = SupportFlow()
flow.chat()
```

Ele cobre o loop local comum:

1. Solicita uma mensagem do usuĂˇrio.
2. Para com `exit` / `quit`, `EOFError` ou `KeyboardInterrupt`.
3. Chama `handle_turn(message, session_id=...)`.
4. Imprime o resultado do assistente.
5. Finaliza traces de sessĂŁo adiados em um bloco `finally`.

Customize o comportamento do terminal com I/O injetĂˇvel:

```python
flow.chat(
    session_id="demo-session",
    prompt="You: ",
    assistant_prefix="Assistant: ",
    exit_commands=("exit", "quit", "bye"),
)
```

Para apps web, workers em background, testes e transportes customizados, continue usando `handle_turn()` diretamente.

### Comportamento customizado do router

Para rodar efeitos colaterais (setup de event bus, telemetria) em toda decisĂŁo de routing, sobrescreva `route_turn`:

```python
class SupportFlow(Flow[ConversationState]):
    conversational = True

    def route_turn(self, context: dict[str, Any]) -> str | None:
        self.event_bus = MyBus(self)
        return super().route_turn(context)
```

Para ignorar o router LLM e escolher uma rota programaticamente, retorne uma string de `route_turn`; retornar `None` cai no `_route_with_config(...)`.

### `append_assistant_message` e `append_agent_result`

Dentro de um handler `@listen(label)`, escolha:

- `self.append_assistant_message(text)` â€” adiciona um turno de assistente visĂ­vel ao usuĂˇrio em `state.messages`. O `converse_turn` do prĂłximo turno vai vĂŞ-lo.
- `self.append_agent_result(agent_name, result, visibility="private")` â€” registra um evento estruturado em `state.events` e uma thread em `state.agent_threads[agent_name]`. Visibilidade pĂşblica tambĂ©m chama `append_assistant_message` automaticamente. Use resultados privados para trabalho de bastidor que nĂŁo deve poluir o histĂłrico canĂ´nico.

`ConversationConfig.visible_agent_outputs` pode promover globalmente os resultados privados de agentes especĂ­ficos para pĂşblicos (`"all"` ou lista de nomes).

## Tracing entre turnos

Com `defer_trace_finalization=True` (padrĂŁo em `ConversationalConfig`):

- **Um batch de trace** para toda a sessĂŁo de chat.
- **`flow_started`** sĂł no primeiro turno; **`flow_finished`** uma vez em `finalize_session_traces()`.
- **`kickoff` por turno** nĂŁo exibe â€śTrace batch finalizedâ€ť.
- **Trabalho aninhado** (`Agent.kickoff()`, crews, tools Exa) acrescenta ao batch **pai**; flows internos de `AgentExecutor` nĂŁo fecham o batch da sessĂŁo cedo.

```python
flow.chat(session_id=session_id)
```

`flow.chat()` chama `finalize_session_traces()` para vocĂŞ. Quando vocĂŞ controla o loop com `handle_turn()` ou `kickoff(...)`, chame `finalize_session_traces()` quando a sessĂŁo terminar.
